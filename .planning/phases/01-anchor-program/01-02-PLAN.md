**open_round instruction:**

This instruction opens the current pixel AND pre-opens the next pixel (N+1) to enable SC-17 future pixel betting. Players can place bets on the next pixel while the current one is still active.

- Accounts struct `OpenRound`:
  - `season_state: Account<'info, SeasonState>` with `mut`, `seeds = [b"season", season_state.season_number.to_le_bytes().as_ref()]`, `bump = season_state.bump`, `constraint = season_state.status == SeasonStatus::Active @ PixelPredictError::SeasonNotActive`
  - `pixel_state: Account<'info, PixelState>` with `init`, `payer = oracle`, `space = 8 + PixelState::INIT_SPACE`, `seeds = [b"pixel", season_state.season_number.to_le_bytes().as_ref(), pixel_index.to_le_bytes().as_ref()]`, `bump`
  - `config: Account<'info, ConfigAccount>` with `seeds = [b"config"]`, `bump = config.bump`, `has_one = oracle @ PixelPredictError::UnauthorizedOracle`
  - `oracle: Signer<'info>` (also payer for PixelState rent)
  - `system_program: Program<'info, System>`

- Arguments: `pixel_index: u16, prompt_hash: [u8; 32]`

- Handler:
  1. Validate `pixel_index == season_state.current_pixel_index` (error: InvalidPixelIndex) -- this ensures sequential opening
  2. Since current_pixel_index tracks the NEXT pixel to open and is only incremented in resolve_round, the check `pixel_index == current_pixel_index` is sufficient for sequential enforcement. No previous pixel account needed.
  3. Derive coordinates: `x = pixel_index % season_state.grid_width as u16`, `y = pixel_index / season_state.grid_width as u16`. Cast to u8.
  4. Set PixelState fields: season_number, pixel_index, x, y, status=Open, color_pools=[0;16], total_pool=0, winning_color=None, shade=None, warmth=None, prompt_hash (from argument), arweave_txid=[0;43], has_arweave_txid=false, vrf_resolved=false, opened_at=Clock::get()?.unix_timestamp, locked_at=None, resolved_at=None, bump=ctx.bumps.pixel_state

**Next-pixel pre-opening (SC-17 enablement):**

After initializing the current pixel, the handler must also create the PixelState for pixel N+1 (if N+1 < total_pixels). This enables players to place bets on the next round before it becomes the active round.

Implementation approach: Use a CPI to System Program to create the next pixel's PDA account within the same instruction. Specifically:
  1. Calculate `next_pixel_index = pixel_index + 1`
  2. Calculate `total_pixels = (season_state.grid_width as u16) * (season_state.grid_height as u16)`
  3. If `next_pixel_index < total_pixels`:
     - Derive the PDA for the next pixel: seeds = `[b"pixel", season_number.to_le_bytes(), next_pixel_index.to_le_bytes()]`
     - Use `anchor_lang::system_program::create_account` CPI to allocate the account with the correct space (8 + PixelState::INIT_SPACE)
     - Initialize the next PixelState fields: season_number, pixel_index=next_pixel_index, x/y derived from next_pixel_index, status=Open, all pools zeroed, prompt_hash=[0;32] (no prompt committed yet for future pixel), opened_at=Clock timestamp, all other fields defaulted
     - The next pixel's prompt_hash will be zeroed (uncommitted) until the oracle explicitly opens it as the current round (at which point it will already exist, so the next open_round call should handle the case where pixel_state already exists)

**Alternative simpler approach (PREFERRED):** Instead of CPI account creation, pass the next pixel's PixelState as an optional account using `Option<Account<'info, PixelState>>` with `init_if_needed`. However, Anchor's `init` doesn't work with `Option` directly. The cleanest approach is:

Create a **separate instruction** `pre_open_next_pixel` that the oracle calls immediately after `open_round`. This avoids complicating the open_round accounts struct. OR, use remaining_accounts.

**SIMPLEST approach (DO THIS):** Modify `open_round` to accept `pixel_index` that can be `current_pixel_index` OR `current_pixel_index + 1`. The constraint becomes: `pixel_index >= season_state.current_pixel_index && pixel_index <= season_state.current_pixel_index + 1`. The oracle calls `open_round` twice in sequence: once for pixel N (current), once for pixel N+1 (next). The second call succeeds because N+1 is within the allowed range. For pixel N+1, the prompt_hash should be `[0u8; 32]` (no prompt committed yet) -- the oracle will update it later when the round actually starts.

Actually, this approach has a problem: `current_pixel_index` only advances at resolve time, so the oracle could also open pixel N+2, N+3, etc. across multiple rounds. Let's constrain more tightly:

**FINAL approach:** Allow `open_round` to accept any `pixel_index` where `pixel_index >= season_state.current_pixel_index` AND `pixel_index <= season_state.current_pixel_index + 1`. This allows opening exactly the current pixel and one pixel ahead. The PDA init constraint ensures each pixel can only be opened once (second init of same PDA fails). The oracle is already trusted (has_one config check), so allowing it to open one pixel ahead is safe.

Update the handler validation:
```rust
require!(
    pixel_index >= season_state.current_pixel_index
        && pixel_index <= season_state.current_pixel_index.checked_add(1).unwrap_or(u16::MAX),
    PixelPredictError::InvalidPixelIndex
);
```

For `pixel_index == current_pixel_index + 1` (the look-ahead pixel), set prompt_hash from the argument (oracle may pass zeroed hash as placeholder). The oracle calls open_round twice per round: once for N (with real prompt_hash), once for N+1 (with zeroed prompt_hash or a pre-committed hash).
