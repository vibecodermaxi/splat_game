<decisions>
## Implementation Decisions

### VRF Provider
- **UPDATED:** Pyth Entropy is EVM-only (no Solana SDK). Claude's discretion to pick between Switchboard Randomness v3 or ORAO VRF based on Anchor 0.32 CPI compatibility
- Full integration in Phase 1 — not stubbed. On-chain VRF proof verification in resolve_round
- If VRF provider itself is unavailable during a fallback attempt, round stays in delayed state — no unverifiable randomness ever enters the system
- No oracle-generated random as a deeper fallback

### Program Security
- Program is upgradeable (standard for early-stage, allows bug fixes)
- Admin key doubles as upgrade authority — single key for both roles
- Oracle keypair stored in a config PDA — admin can rotate oracle key without program redeployment
- Treasury and jackpot wallet addresses hardcoded as program constants — not admin-redirectable, even if admin key is compromised
- This split (rotatable oracle, immutable fund destinations) balances operational flexibility with fund security

### Prompt Data Storage
- Full prompt text stored off-chain on Arweave (permanent, decentralized, ~$0.001/upload)
- Only the SHA-256 hash stored on-chain in PixelState (pre-committed at open_round)
- Arweave transaction ID stored on-chain after upload (in PixelState or a lightweight reference)
- No on-chain hash verification at resolve time — verification is off-chain: fetch from Arweave, hash, compare to committed hash
- resolve_round does NOT receive prompt_data as an argument (spec change from Vec<u8>)

### Network & Testing Strategy
- Development on local validator (solana-test-validator)
- Deploy to devnet when tests pass — mainnet deployment deferred to later phase
- Full test suite required:
  - Rust unit tests for payout math, validation logic, edge cases (truncation, overflow, zero-winner)
  - TypeScript integration tests for every instruction + full round lifecycle flows
  - Edge case coverage: zero-winner rounds, max bet limits, lockout timing, VRF resolution path, season completion

### Claude's Discretion
- Account sizing and rent allocation strategy
- PDA seed structure (spec provides suggestions but implementation may optimize)
- Error code numbering and naming
- Instruction ordering within transactions
- Whether to use Anchor events for indexing

</decisions>

<specifics>
## Specific Ideas

- The spec defines account structures in detail (SeasonState, PixelState, BetAccount, PlayerSeasonStats) with exact field types — use these as the starting point
- Payout math must use u128 intermediates and multiply-before-divide (from research: SPL token-lending $27M/hr vulnerability was this class of bug)
- Zero-winner branch in resolve_round must transfer net pool (total_pool × 95%) to treasury — spec FAQ mentions this but no instruction handles it
- correct_predictions updated at claim time, not resolve time (spec note: avoids unbounded compute at resolution)

</specifics>
