# Roadmap: Pixel Predict

## Overview

Build a Solana parimutuel prediction market where players bet on an AI artist's color choices. The dependency chain is strict — Anchor program first (everything else reads its IDL), oracle second (drives the game clock), frontend core third (the betting loop), trust layer last (verifiability, season completion, docs). Four phases map cleanly to the four independent subsystems.

## Phase Details

### Phase 1: Anchor Program
**Goal**: A deployed, audited Anchor program on devnet (and mainnet) that owns fund custody, enforces all betting rules, and runs the complete round lifecycle
**Depends on**: Nothing (first phase)
**Requirements**: SC-01, SC-02, SC-03, SC-04, SC-05, SC-06, SC-07, SC-08, SC-09, SC-10, SC-11, SC-12, SC-13, SC-14, SC-15, SC-16, SC-17
**Success Criteria** (what must be TRUE):
  1. Admin can initialize a season and the first round can be opened with a commit hash on-chain
  2. Player can place, and increase, a bet on one color per pixel within min/max constraints — and a second bet on the same pixel is rejected on-chain
  3. Round resolves correctly: winning bets receive proportional payout (with 5% rake split to treasury/jackpot), zero-winner rounds transfer net pool to treasury
  4. Player can claim winnings and unclaimed winnings persist; season auto-completes at pixel 100
  5. All payout arithmetic uses u128 intermediates and multiply-before-divide — a 1-lamport bet against a large pool does not truncate to zero
**Plans:** 6/6 plans complete

### Phase 2: Oracle Service
**Goal**: A running Railway process that autonomously drives the round lifecycle — opening rounds with committed hashes, calling Claude at temperature 0, resolving on-chain, and failing over to VRF — with no human intervention needed
**Depends on**: Phase 1
**Requirements**: ORC-01, ORC-02, ORC-03, ORC-04, ORC-05, ORC-06, ORC-07, ORC-08, ORC-09, ORC-10, ORC-11
**Success Criteria** (what must be TRUE):
  1. Oracle opens a round by posting SHA-256 prompt hash on-chain before the betting window
  2. Oracle calls Claude at temperature 0, parses COLOR/SHADE/WARMTH/REASONING, validates the response, and resolves the round on-chain with the full prompt data
  3. On Claude API failure: oracle retries 3x at 30 seconds, then enters 5-minute retry loop for 30 minutes, then falls back to VRF resolution — round always resolves
  4. Oracle runs as an always-on Railway process (not cron) and maintains round_history.json across round boundaries

### Phase 3: Frontend Core
**Goal**: A playable, mobile-first game loop where a player with a connected wallet can view the live canvas, place a bet, watch the countdown, see the round resolve with animation, and claim winnings
**Depends on**: Phase 2
**Requirements**: FE-01, FE-02, FE-03, FE-04, FE-05, FE-06, FE-07, FE-08, FE-09, FE-10, FE-11, FE-12, FE-13, FE-14, FE-15, FE-16
**Success Criteria** (what must be TRUE):
  1. Player connects a Phantom, Solflare, or Backpack wallet and sees the live 10x10 canvas with shade/warmth-modified pixel colors and a pulsing active pixel
  2. Player selects a color, enters a bet amount (or uses quick-set buttons), submits the transaction, and receives confirmation feedback — bet button shows "Bets locked" in the final 2 minutes
  3. Countdown timer tracks the on-chain opened_at timestamp; pool percentages and multipliers refresh every 60 seconds from on-chain data
  4. Round resolution triggers a color flood animation with a win or loss notification; canvas and season progress indicator (round N of 100) update immediately
  5. All interactions work on a 375px-wide mobile screen with thumb-reach tap targets; dark video-game-casual aesthetic renders correctly

### Phase 4: Trust Layer and Docs
**Goal**: A complete, shippable product where players can verify the AI's input was committed before they bet, see season stats, claim from My Bets, share completed canvases, and read clear docs — ready for mainnet launch
**Depends on**: Phase 3
**Requirements**: FS-01, FS-02, FS-03, FS-04, FS-05, FS-06, FS-07, FS-08, FS-09, FS-10, FS-11, FS-12, FS-13, DOC-01, DOC-02, DOC-03, DOC-04, DOC-05
**Success Criteria** (what must be TRUE):
  1. My Bets panel shows active, resolved, and claimable bets; player can claim individual or all winnings; personal season stats (volume, hit rate) are visible
  2. Each resolved pixel shows commit-reveal data (prompt hash + published prompt); VRF-resolved pixels have a visible visual distinction on canvas; tapping a pixel shows color, shade, warmth, and round number
  3. Season completion triggers a full-screen canvas; player can generate and download a branded PNG ("Pixel Predict — Season X"); 12-hour intermission shows countdown and season stats
  4. Jackpot balance is visible with "Coming soon" label; failed transactions surface a clear error with retry option; wallet disconnect shows recovery state
  5. Docs site covers How It Works, The AI Artist, Betting Rules, Fairness & Verification, and FAQ — all commit-reveal copy uses "prompt commitment proof" language, not "reproducible result" language
