# Requirements: Pixel Predict

**Defined:** 2026-03-17
**Core Value:** Players can place real-money bets on AI color choices with immediate, verifiable, on-chain resolution

## v1 Requirements

### Smart Contract

- [x] **SC-01**: Admin can initialize a new season with configurable grid dimensions
- [x] **SC-02**: Oracle can open a round by posting SHA-256 prompt hash on-chain
- [x] **SC-03**: Player can place a bet on one of 16 colors for an open pixel round
- [x] **SC-04**: Player can increase their bet on their chosen color before lockout
- [x] **SC-05**: Player cannot bet on more than one color per pixel
- [x] **SC-06**: Minimum bet of 0.01 SOL and maximum of 10 SOL per color per pixel enforced on-chain
- [x] **SC-07**: Anyone can trigger lock_round after 28-minute betting window expires (permissionless crank)
- [x] **SC-08**: Oracle can resolve a round by posting winning color, shade, warmth, and prompt data
- [x] **SC-09**: Rake of 5% deducted from pool at resolution (3% treasury, 2% jackpot) transferred atomically
- [x] **SC-10**: Player can claim proportional payout from winning pool after round resolution
- [x] **SC-11**: Unclaimed winnings persist indefinitely
- [x] **SC-12**: Payout math uses u128 intermediates and multiply-before-divide to prevent truncation
- [x] **SC-13**: Zero-winner rounds transfer net pool to treasury wallet
- [x] **SC-14**: Season completes automatically when final pixel resolves (current_pixel_index == grid_width * grid_height)
- [x] **SC-15**: Player season stats tracked on-chain (total bets, volume, correct predictions updated at claim)
- [x] **SC-16**: VRF fallback resolution supported with vrf_resolved flag on PixelState
- [x] **SC-17**: Player can place bets on future pixels before their round opens

### Oracle Service

- [x] **ORC-01**: Oracle constructs AI prompt from current canvas state, neighbors, and last 5 selections
- [x] **ORC-02**: Oracle computes SHA-256 hash of prompt and posts it on-chain at round open
- [x] **ORC-03**: Oracle calls Claude API at temperature 0 and parses COLOR, SHADE, WARMTH, REASONING
- [x] **ORC-04**: Oracle validates AI response (color is one of 16, shade/warmth 0-100)
- [x] **ORC-05**: Oracle posts winning color, shade, warmth, and full prompt data on-chain at resolution
- [x] **ORC-06**: Oracle retries failed API calls 3 times at 30-second intervals
- [x] **ORC-07**: Oracle enters delay state on persistent failure, retrying every 5 minutes for 30 minutes
- [x] **ORC-08**: Oracle falls back to VRF resolution after 30 minutes of failure
- [x] **ORC-09**: Oracle maintains local round_history.json for prompt construction (last 5 rounds)
- [x] **ORC-10**: Oracle runs as always-on Railway process with in-process scheduler (not Railway cron)
- [x] **ORC-11**: Oracle includes season style summary in system prompt starting from Season 2

### Frontend — Game Loop

- [x] **FE-01**: User can connect Solana wallet (Phantom, Solflare, Backpack)
- [x] **FE-02**: User sees 10×10 canvas with filled pixels showing shade/warmth-modified colors
- [x] **FE-03**: Active pixel has pulsing highlight/border animation
- [x] **FE-04**: User sees 16-color betting panel with pool percentages and payout multipliers
- [x] **FE-05**: User can select a color and enter bet amount with quick-set buttons (0.01/0.05/0.1/0.5/1 SOL)
- [x] **FE-06**: User sees transaction confirmation feedback after placing bet
- [x] **FE-07**: User sees countdown timer synced to on-chain opened_at timestamp
- [x] **FE-08**: Timer shows visual shift in final 2 minutes (lockout period)
- [x] **FE-09**: Bet button disabled with "Bets locked" label during lockout
- [x] **FE-10**: User sees resolution animation when round resolves (color flood + win/loss notification)
- [x] **FE-11**: Pool distribution updates from on-chain data every 60 seconds
- [x] **FE-12**: Frontend uses Helius WebSocket subscriptions for live state updates
- [x] **FE-13**: Frontend falls back to polling every 15 seconds if WebSocket disconnects
- [x] **FE-14**: All interactions work with thumb-reach tap targets on mobile (375px+ width)
- [x] **FE-15**: Dark background with video-game-casual aesthetic (rounded corners, chunky buttons, playful typography)
- [x] **FE-16**: User sees season progress indicator (round N of 100)

### Frontend — Supporting

- [x] **FS-01**: User can view active bets, resolved bets (won/lost), and claimable winnings in My Bets panel
- [x] **FS-02**: User can claim individual winnings or claim all from My Bets panel
- [x] **FS-03**: User sees personal season stats (total bet, total won, correct predictions, hit rate)
- [x] **FS-04**: User sees season completion screen with full-screen canvas when final pixel resolves
- [x] **FS-05**: User can generate and share branded PNG of completed canvas ("Pixel Predict — Season X")
- [x] **FS-06**: User sees 12-hour intermission countdown with season stats after completion
- [x] **FS-07**: VRF-resolved pixels display visual distinction on canvas (border or indicator)
- [x] **FS-08**: User can tap filled pixel to see result details (color, shade, warmth, round number)
- [x] **FS-09**: Commit-reveal information displayed (prompt hash + published prompt accessible per round)
- [x] **FS-10**: How to Play in-app explainer (what is it, how to bet, how odds work, how AI picks, fairness)
- [x] **FS-11**: Jackpot wallet balance displayed with "Coming soon" teaser
- [x] **FS-12**: Failed transaction errors surfaced with clear message and retry option
- [x] **FS-13**: User sees error state and recovery when wallet disconnects mid-session

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time pool updates (sub-second) | Enables last-second copycat sniping; 60s cadence is intentional |
| Refunds on round failure | DDoS-for-refund attack vector; rounds always resolve via VRF fallback |
| Order book / limit orders | Requires market makers and liquidity; kills casual accessibility |
| Multiple colors per pixel per player | Eliminates commitment mechanic that drives engagement |
| In-app chat / social feed | High moderation burden; crypto chat = spam/scam vectors |
| Admin dashboard (v1) | Expands attack surface; CLI-based management sufficient for single operator |
